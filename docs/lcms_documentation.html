1. System Overview

The Legal Case Management System (LCMS) centralises the end-to-end processing of litigation cases for the organisation. It supports multiple litigation categories, streamlined appeal/cassation stages, robust supervision, granular audit logging and real-time notifications. LCMS is built on Laravel 12, MySQL and Tailwind-powered Blade front-end templates.

3. Case Lifecycle Workflow

  1. Case Creation – Lawyer fills entry form, plaintiffs/defendants, claimed amount.
  2. Direct Suit Stage – Court filings, hearings, progress updates.
  3. Appeal Stage – If marked “Appeal Filed”, new appeal record opens.
  4. Second Appeal Stage – Triggered strictly after Appeal decision.
  5. Cassation Stage – Final appellate review; post-decision either execution or closure.
  6. Execution – Asset recovery / judgment enforcement where decision favourable.
  7. Final Closure – Supervisor validates and marks case closed.

4. Detailed Workflows

4.1 Supervisor Approval Workflow

After Cassation verdict, case status changes to Pending Execution. Supervisor Panel lists waiting cases:

  • Supervisor reviews decision summary & attachments.
  • Action buttons: Approve Execution / Request Follow-Up.
  • Approval auto-creates a progress update “Execution Approved” and sets status Execution.

4.2 Appeal / Cassation Workflow

StageTriggerEditable FieldsStatus Updates Generated
Appeal Lawyer ticks “Appeal Filed” on suit decision form File #, Court, Notes, Decision Date, Claimed Amount "Appeal Filed" → "Appeal Hearing Scheduled" → "Appeal Decision Passed"
Second Appeal Appeal decision recorded & lawyer selects "Second Appeal Filed" Same as Appeal "Second Appeal Filed" → "Second Appeal Hearing" → "Second Appeal Decision"
Cassation Second Appeal verdict stored and lawyer files Cassation Same as Appeal "Cassation Requested" → "Cassation Verdict"

4.3 Notification System

  • Email + in-app banner notifications for: new case assignment, upcoming hearing (24h prior), supervisor decisions.
  • Notifications stored in notifications table with polymorphic link to notifiable.

4.4 Audit Logging

Every create/update/delete on critical tables triggers AuditLog entry (model trait Auditable):

  • Stored fields: table, record_id, action, user_id, values_before, values_after, ip.
  • Admin view: System > Logs with CSV export.
